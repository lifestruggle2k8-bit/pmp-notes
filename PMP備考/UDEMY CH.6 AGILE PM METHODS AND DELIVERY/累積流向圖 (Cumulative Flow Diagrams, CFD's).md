# 累積流向圖 (Cumulative Flow Diagrams, CFD's) 終極識別指南

## 一、 CFD 的圖表結構與動態進展

**累積流向圖（Cumulative Flow Diagrams, CFD's）** 是一種強大的**堆疊圖表 (Stacked graphs)**，在敏捷專案中主要用於高度視覺化呈現專案工作的進展過程、已完成的點數 (Points completed)/工作量，以及各個特定階段的工作量分布。

### 1. 圖表基本維度

- **縱軸 (Y軸)**：代表工作量（例如專案起始的總工作量點數，如 600 點）。
    
- **橫軸 (X軸)**：代表專案推進的時間線。
    
- **堆疊區域**：透過不同顏色的堆疊區塊，清晰區分出各個相依的工作狀態。常見的標準流動順序為：
    
    $$\text{Ready (就緒)} \longrightarrow \text{Dev (開發)} \longrightarrow \text{Test (測試)} \longrightarrow \text{Deployed (已部署)}$$
    
![[screenshot-01KSATVYH73GX2QVPGT717PTG5.png]]
### 2. 工作流的動態特性

隨著專案時程向前推進，團隊會持續「切入」工作量並完成它，使總未完工工作量逐漸減少。工作卡片會依序在各個顏色區域之間流動，而各區域的「垂直寬度」則誠實反映了該階段目前所累積、滯留的在製品（WIP）工作量。

## 二、 核心考點：CFD 與限制理論、利特爾法則的關聯

CFD 絕非單純的進度表，它是診斷團隊生產力系統的臨床 X 光片，與以下核心管理理論高度綁定：

- **利特爾法則 (Little's Law) 的應用**：
    
    依據「週期時間與隊列長度成正比」的原理，管理者**僅需拉近視線、觀察 CFD 圖表某一時間點特定狀態的「水平寬度（隊列 Queue）」**，即可精確判斷與預測該批工作完成所需的提前期（Lead Time）。
    
- **瓶頸與限制理論 (Theory of Constraints, TOC)**：
    
    這是認證考試中高頻出現的核心概念。當系統中某個工作流在特定階段停滯不前，代表系統中存在限制（Constraint）。
    

## 三、 必考大魔王：如何透過 CFD 視覺特徵「識別瓶頸」？

在備考情境題時，考生常被圖表中的「變寬區域」誤導。請務必釐清以下互為因果的**深層診斷邏輯**：

```
 📊 CFD 瓶頸診斷邏輯鏈：
 
 ┌────────────────────────┐        ┌────────────────────────┐
 │  下游某階段能力不足    │ ───►   │  上游完工的任務卡住    │
 │ (呈現【平坦】的瓶頸活動)│        │  無法流入下一階段      │
 └────────────────────────┘        └────────────────────────┘
                                               │
                                               ▼
                                   ┌────────────────────────┐
                                   │  導致上游區域被迫積壓  │
                                   │  在圖表上呈現【變寬】  │
                                   └────────────────────────┘
```
![[screenshot-01KSATVYH73GX2QVPGT717PTG5.png]]
### 1. 變寬的活動區域 (Widening Area Activity) ─── 瓶頸造成的「結果」

- **視覺特徵**：圖表中某個顏色區域（例如紅色的 Dev 區域）隨著時間軸推移，垂直寬度變得**越來越寬**。
    
- **背後含意**：這代表工作正在此處發生嚴重的**堆積與堵塞**，無法順利流向下一階段。但請注意，**這個變寬的區域本身，並不是罪魁禍首，它是被下游卡死而產生的「受害者（結果）」**。
    
![[screenshot-01KSATVYH73GX2QVPGT717PTG5.png]]
### 2. 瓶頸活動 (Bottleneck Activity) ─── 造成變寬的「根本原因」

- **視覺特徵**：在變寬區域的緊鄰下游，呈現**平坦、沒有變寬、甚至增長停滞**的活動區域。
    
- **背後含意**：因為該下游階段的實際處理能力（Capacity）嚴重不足，無法消化更多工作。
    
![[screenshot-01KSATVYH73GX2QVPGT717PTG5.png]]
### 3. ⚖️ 實例診斷分析（以 Dev 變寬、Test 平坦為例）

敏捷開發本質上是一個「拉動系統 (Pull System)」。當下游階段（瓶頸）無法處理更多工作時，它就不會主動向開發階段「拉取」新工作。

- **現象**：若 CFD 上 **Dev（開發）區域不斷變寬**，而下游的 **Deployed（部署）產出曲線卻變為水平或停滯**。
    
- **診斷**：**Test（測試）階段就是真正的系統瓶頸 (Bottleneck)！**
    
- **原因解讀**：因為測試人員產能觸頂，無法處理更多工作，導致開發人員雖然快速寫完程式碼，卻**無法將工作推入測試環境**。開發完的工作被迫全部卡在開發階段，進而在 CFD 圖表上導致上游的 Dev 區域呈現出不斷被撐開、變寬的積壓現象。
    
![[screenshot-01KSATVYH73GX2QVPGT717PTG5.png]]
## 💡 三點需要牢記的部分

1. **CFD 區域變寬代表「工作堆積」，是瓶頸造成的受害結果**：在備考看圖題時，這是最容易踩中的直覺陷阱。**區域變寬的階段（如 Dev 變寬）絕對不是瓶頸活動**，它只是代表該階段的工作在「排隊等待」。真正卡住大家的「瓶頸活動」，位於這個變寬區域的**緊鄰下游階段（如 Test）**。
    
2. **下游「平坦不增長」搭配上游「持續堆積」是鎖定瓶頸的終極鐵證**：診斷系統障礙時，必須同時觀察上下游。若上游（Dev/Test）持續有工作進入且區域外擴，但最下游（Deployed）的曲線卻呈現死寂的**水平狀態**，這表示工作在中間被某個限制（Constraint）徹底截斷，必須立刻調配僕人式領導的資源去移除該下游障礙。

3. **依據利特爾法則，CFD 的寬度就是專案曝險的 WIP 規模**：CFD 完美的將 Kanban 限制 WIP 的精神視覺化。圖表上堆疊區域的垂直高度就是當下的在製品（WIP）總量。WIP 越寬，代表隱藏的瓶頸越多、潛在的系統風險（如代碼損壞、規格脫節）就越大。團隊應以收窄各區域寬度、讓線條平行向上滑升為最優流動目標。
