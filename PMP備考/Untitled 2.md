## Planning Poker

- 一種在 Agile 專案中非常常見的遊戲，用於為 User Stories 指派點數
- **核心特性**
    - **匿名性**：過程傾向於匿名進行
    - **協作性**：高度互動且具備協作性質
- **估算方式**
    - 使用印有**費波那契數列 (Fibonacci sequence)** 的卡片進行估算

### Planning Poker 的執行流程

- **準備階段**
    - 每位參與者都會獲得一組印有費波那契數列數字的卡片
    - 使用的數列包含：`1, 2, 3, 5, 8, 13, 21` 等
- **開始估算**
    - 由會議主持人（Facilitator，例如 Product Owner）讀出一個 User Story
    - 參與者根據讀出的內容，從手中的卡片中選擇合適的數字

### User Story 範例與思考時間

- **User Story 範例內容**
    - 「作為一名應收帳款人員（Accounts Receivable Personnel），我希望能輸入發票（Invoice），以便我能準時收到款項，或讓客戶知道他們欠我們多少錢。」
- **思考階段**
    - 主持人讀完 User Story 後，會留出時間讓團隊思考
    - **思考時間長短**
        - 可能僅需 30 秒，也可能需要幾分鐘
        - 具體時間取決於**團隊動態 (Team Dynamics)**

### 估算與揭曉過程

- **思考階段**
    - 團隊成員在思考時會保持沉默
    - 每個成員會在心中決定要為該 User Story 指派多少點數（例如：3、5 或 8 點）
- **確保匿名性 (Anonymity)**
    - 主持人會指示成員在思考完成後，各自從手中選出一張卡片放下
    - **關鍵動作**：放下卡片時**不要翻轉數字**
        - 這樣可以確保在揭曉數字前，沒有人知道其他人的估算值
        - 這種做法與 **Wideband Delphi** 方法非常相似

### 揭曉估算值與達成共識

- **同步揭曉過程**
    - 主持人會設置計時器並進行倒數（例如從 5 秒倒數至 0）
    - 在倒數結束時，所有成員同時翻轉卡片，讓數字公開
- **處理估算分歧**
    - 揭曉後會觀察團隊成員選擇數字的分佈情況
    - **範例情境**：假設團隊有 8 位成員
        - 有 5 位成員選擇 `5` 點
        - 有 2 位成員選擇 `3` 點
        - 有 1 位成員選擇 `8` 點
    - **決策方式**：由於多數成員（Majority）都選擇了 `5`，因此該項功能的估算值會記錄為 `5`

### 處理缺乏共識的情況

- 當多數決無法達成明確共識時的應對策略
    - **情境範例**：假設團隊對於第二個 User Story 的估算結果如下：
        - 3 位成員選擇 `5` 點
        - 3 位成員選擇 `8` 點
        - 2 位成員選擇 `3` 點
    - **處理方式**：由於點數分散在 3、5、8 之間，沒有明顯的多數，此時應採取最保守的做法
        - **選擇最高點數**：直接採用該組數字中的最高值（在此範例中為 `8` 點）
        - **目的**：確保在估算時預留足夠的緩衝空間，避免低估開發工作量
- **後續流程**
    - 一旦該 User Story 的點數確定後，即可進入下一個 User Story 的估算循環

### Planning Poker 的特性與優勢

- **高參與度**
    - 是一種互動性極高的方法，能有效帶動團隊成員參與討論
- **執行特性**
    - **低技術門檻 (Low tech)**：不需要複雜的軟體或設備
    - **高互動性 (High touch)**：強調成員間的直接溝通與接觸
- **適用場景**
    - 非常適合在 Agile 專案的規劃會議中使用
