## 交付績效領域

- 與**工作績效領域**不同，後者管理工作執行（如確保牆壁塗漆或程式碼撰寫正確）。
- 聚焦最終階段：將工作交付給利害關係人，並確認專案是否達成主要目標。
- 處理與**交付專案所承擔的範疇與品質**相關的活動與功能。
    - 專案交付在於滿足**需求、範疇與品質期望**，以產生預期的交付項目。
- **成果**：
    - 專案有助於達成業務目標。
    - 專案實現成果。
    - 專案效益在時限內實現。
    - 專案團隊對需求有理解。
    - 利害關係人接受並對專案交付項目感到滿意。

## 交付驗證範例

- 確認是否滿足**需求、範疇與品質期望**。
- 例如：油漆房間時，檢查是否達到藍色品質期望。
- 有些專案在**整個過程中**交付價值，有些則在**最後集中交付**。
- 專案經理需理解**交付物如何為利害關係人交付價值**。
    - **需求蒐集**。
    - **演進與發掘需求**。
    - **管理需求**。
    - **定義與分解範疇**。
    - **交付物的完成**。

## 品質要素與成本

- **品質需求**反映在：
    - 完成標準。
    - 完成定義。
    - 工作說明。
    - 需求文件。
- 專案經理須考慮**品質成本**：
    - **預防**。
    - **評估**。
    - **內部失敗**。
    - **外部失敗**。
    - **變更成本**。
- 交付最終產品給客戶，即使符合範疇，也需滿足**品質期望**。

## 品質成本類型

- **品質成本**：預防、評估、內部失敗、外部失敗、變更成本。
- **預防成本**：聘請更有經驗木匠裝修房間。
- **內部失敗成本**：未達品質需返工或更改產品。
- **變更成本**：專案進展中改變需求，越晚越貴。

### 交付績效領域的衡量方式

- **目標檢查**：驗證專案是否達成領域成果。
    - **專案貢獻業務目標**：組織計畫與專案交付項目對齊。
    - **實現啟動成果**：業務案例與基礎數據顯示專案按軌道前進。
    - **效益按時實現**：交付項目如計畫般達成。
    - **團隊理解需求**：預測開發中初始需求少變反映理解；需求演進專案中，理解可能較晚形成。
    - **利害關係人滿意**：訪談、觀察及終端使用者反饋顯示滿意度。

| 成果 | 檢查方式 |
| --- | --- |
| 專案貢獻業務目標與策略推進 | 組織計畫顯示交付項目與業務目標對齊 |
| 實現啟動交付成果 | 業務案例與數據顯示按軌道實現預期成果 |
| 效益如計畫時限實現 | 交付如計畫達成 |
| 團隊清楚理解需求 | 預測開發：初始需求少變；需求演進：理解可能較晚 |
| 利害關係人接受並滿意交付項目 | 訪談、觀察、使用者反饋顯示滿意 |

- **團隊清楚理解需求**的檢查方式：
    - **預測性開發**（predictive）：初始需求**少變動**反映理解良好；若變動多，表示團隊可能不理解。
    - **需求演進專案**：清楚理解需求可能要到專案**較晚階段**才形成。
- **交付績效領域的核心檢視**：專案結束時，確認是否達成**主要目標**。
    - 例如：組織想**增加營收**，是否做到？
    - 想**減少開支**，是否做到？
- 這領域聚焦**專案是否交付價值**，極為重要。
