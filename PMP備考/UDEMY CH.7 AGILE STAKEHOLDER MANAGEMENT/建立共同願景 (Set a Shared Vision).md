# 建立共同願景 (Set a Shared Vision) 核心指南

在敏捷專案管理中，**建立共同願景（Set a Shared Vision）** 是奠定專案成功的底頭地基。它的核心目標是確保客戶與敏捷專案團隊大腦中的藍圖完全同步，以便在最終交付產品時，能精準無誤地符合客戶的實際需求。

## 一、 共同願景的雙向對齊與信任邏輯

建立共同願景不僅僅是單向的需求收集，而是一場關於**信任**的心理對齊：

- **雙向理解鐵律**：在溝通過程中，團隊必須確保做到兩大底線：
    
    1. **你（團隊）深刻了解客戶的需求。**
        
    2. **客戶也百分之百知道「你了解他們的需求」。**
        
- **信心的黃金三角邏輯**：
    
    $$\text{團隊知道客戶要什麼} \longrightarrow \text{客戶確信團隊知道他們要什麼} \longrightarrow \text{建立牢固的客戶信心與信任}$$
    
    當客戶內心確信「團隊完全了解我的業務藍圖」時，他們才會對專案產生實質的安全感，並相信產品最終完工時定能符合其預期。
    

## 二、 消除溝通落差帶來的負面心理焦慮

若專案在前期或日常中缺乏共同願景的建立，客戶在溝通結束後極易陷入嚴重的內耗與不安，進而引發專案的信任危機：

### 🚨 缺乏共同願景時的客戶負面心理

- **強烈的不信任感**：客戶會在心中暗自質疑開發者或承包商的專業度，產生「這個人根本不懂我（He doesn't understand me）」的排斥心理。
    
- **焦慮與自我懷疑**：客戶離開會議後會不斷反思是否自己當初沒表達清楚，進而引發對專案會搞砸的巨大恐懼（「我可能沒解釋清楚，他們之後會搞砸 I don't think I explained that enough」）。
    

> 🎯 **敏捷的核心目標**：透過持續對齊願景，**徹底消除客戶這種深層的「不安感」與疑慮**。轉而追求正面的心理感受——讓客戶在內心深處產生強烈的信心，確信團隊與其腦海中構思的藍圖（Envisioning）高度同步。

## 三、 實務中建立共同願景的七大核心工具

敏捷不依賴厚重的合約文字來堆砌願景，而是推薦使用以下七種具備**高視覺化、高互動性、以及高共識性**的具體方法來落實共同願景：

```
 🛠️ 建立共同願景 (Shared Vision) 的七大工具箱：
 
 ├── 1. Agile Charter (敏捷章程) ───► 高層級預算、5W1H 授權與執行共識協議
 ├── 2. Definition of "Done" (DoD) ─► 團隊與利害關係人對品質與完工狀態的共享願景
 └── 3. Agile Modeling (敏捷建模) ──► 透過視覺化圖表與設計將抽象需求具體化：
         ├── ① 使用案例圖 (Use Case Diagram) ─► 定義系統邊界與使用者互動行為
         ├── ② 資料模型 (Data Models) ─────► 理清底層核心資料結構與邏輯關係
         ├── ③ 畫面設計 (Screen Design) ────► 具象化前端操作體驗與介面編排
         ├── ④ 線框圖 (Wireframes) ────────► 快速拼湊系統骨架與佈局流程
         └── ⑤ 人物誌 (Personas) ──────────► 形塑出具體、鮮明的目標使用者畫像
```

1. **Agile Charter (敏捷章程)**：
    
    作為開發團隊、專案經理與組織高層之間的共識協議，釐清 5W1H 核心，建立對專案「如何進行」的宏觀願景。
    
2. **Definition of "Done" (完成的定義, DoD)**：
    
    針對使用者故事、版本發布與交付物制定統一的品質門檻，建立團隊與客戶對「完工狀態」的共享願景，消滅交付時的負面驚訝。
    
3. **Agile Modeling (敏捷建模) 家族系列**：
    
    透過視覺化模型將無形、抽象的知識需求實體化，是跨越「評估鴻溝」的最強武器：
    
    - **Use case diagram (使用案例圖)**：清晰定義系統的角色、邊界，以及彼此之間的互動行為。
        
    - **Data models (資料模型)**：梳理系統底層核心數據的結構與逻辑關係。
        
    - **Screen design (畫面設計)**：提供直觀的視覺外觀，讓客戶對系統的未來長相一目了然。
        
    - **Wireframes (線框圖)**：以低保真度的骨架結構，高效率地與客戶核對功能佈局與操作流程。
        
    - **Personas (人物誌)**：形塑出栩栩如生的虛擬目標用戶畫像，協助團隊與客戶站在同一個使用者的視角來審視產品價值。
        

## 💡 三點需要牢記的部分

1. **共同願景的核心是消除客戶的「內心不安」，而非填寫流程表單**：在備考情境題時，若遇到客戶表現出「擔心團隊做錯」、「懷疑自己沒說清楚」等焦慮心理時，最核心的敏捷解法就是引導團隊**建立共同願景（Set a Shared Vision）**。必須做到「客戶知道你懂他」，才能從心理學層面斬斷信任危機，讓客戶建立起產品會符合預期的強烈信心。
    
2. **敏捷建模（Agile Modeling）是跨越評估鴻溝的視覺化重炮**：當人類語言的口頭描述發生失真時，團隊應立即祭出敏捷建模工具。不論是能夠釐清行為的**使用案例圖**、釐清邏輯的**資料模型**，還是能讓客戶直接看到半成品的**線框圖（Wireframes）**與**畫面設計**，這些實體視覺媒介能讓客戶腦中的幻影落地，確保雙方在幾分鐘內完成認知校準。
    
3. **章程與 DoD 是制度化的願景對齊，一個管啟動、一個管完工**：必須將 **Agile Charter** 與 **DoD** 視為建立共同願景的兩大制度化支柱。專案章程在期初用 140 字的 Project Tweet 和 How 協議對齊組織出發的方向；完成的定義（DoD）則在迭代中對齊完工與品質的邊界。兩者雙劍合璧，才能確保專案從開局到結案，團隊與客戶的想法始終高度同步。
