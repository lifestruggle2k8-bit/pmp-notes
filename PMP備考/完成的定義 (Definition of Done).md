# 完成的定義 (Definition of "Done", DoD) 核心指南

在敏捷專案管理中，**完成的定義（Definition of "Done", DoD）** 絕非一項單純的技術檢查清單，其核心本質在於**建立對「完成」狀態的共享願景（Creating a shared vision of what done looks like）**，確保團隊與利害關係人對產出品質有著完全一致的認知。

## 一、 為什麼需要頻繁對齊「完成的定義」？

人類的日常溝通天生充滿不確定性，專案團隊必須透過頻繁討論 DoD，來防範以下三大致命的**認知落差**：

- **聽者的主觀誤解**：聽者（開發團隊）根據描述在腦中建立的視覺化想像（Vision），可能與說話者（客戶/PO）實際想要的東西完全不同。
    
- **描述與意圖的不一致**：說話者所口頭描述出來的規格，往往並非他們內心真正想要的東西（_範例_：客戶要求「這個廚房碗碟」，但其講出的細節與心中真正的需求並不匹配）。
    
- **想法隨時間動態改變**：市場在變，利害關係人的想法與預期也會隨著時間推移而發生實質位移。
    

### 🎯 必須全面制定 DoD 的三大對象層次

完成的定義不能只停留在程式碼層面，必須由微觀到宏觀，針對以下三個層次進行明確制定：

1. **使用者故事 (User Stories)**：確保單一功能的驗收準則與品質門檻在開發前就已對齊。
    
2. **版本發布 (Releases)**：確保產品增量在打包推向市場時，符合安全性、法規與部署標準。
    
3. **最終專案交付物 (Final Project Deliverables)**：確保整個專案在法律與商業層面的終極完工底線。
    

## 二、 建立正確定義的終極目標：消除交付意外

- **核心目標**：正確建立「完成的定義」，以便將產品開發到最健康、最精準的狀態，使其完美符合客戶的真實需求。
    
- **消除交付時的意外（No Surprises）**：當軟體開發完成並正式交付時，**使用者在查看產品時不應感到任何驚訝或意外**。
    
- **拒絕錯位反應**：卓越的 DoD 能夠完全避免使用者看到成品後，說出：「喔，我不是要這個」之類的負面反應。
    

## 三、 實務診斷：如何辨識 DoD 的失效？

敏捷專案經理與產品負責人可以透過交付現場的「使用者表情」，來直接臨床診斷專案的 DoD 機制是否健康：

```
 🚨 交付現場的表情診斷學：
 
 【😐 使用者面無表情 / 理所當然】 ──► 完美！完全符合預期，表示 DoD 建立極其正確。
                                                     
 【😳 使用者困惑 / 驚訝：「哇，我沒預料到這個」】 ──► 警訊！表示「完成的定義」期初未對齊，
                                                     溝通已發生結構性斷裂。
```

- **致命的警訊**：若使用者在查看軟體成品時，表現出困惑的表情，或是脫口說出：「哇，我沒預料到這個」，**這代表「完成的定義」在前期根本沒有被正確建立**。
    
- **後續代價**：這種交付時的負面意外會導致嚴重的重工與信任危機，因為產出的產品本質上並未符合用戶內心的預期。
    
- **使用者故事的黃金防線**：因此，在撰寫與討論使用者故事（User Stories）的階段，必須**確保定義已正確完成，且全體相關人員（PO、開發者、測試者）都深刻理解該定義**，這是維持團隊開發方向始終不偏離航道的黃金防線。
    

## 💡 三點需要牢記的部分

1. **DoD 是消滅「我以為」的共享願景，必須具象化且頻繁核對**：在備考與實務中，必須牢記 DoD 的靈魂是 **Shared vision**。因為人類語言存在描述與意圖的天然落差（如廚房碗碟案例），不能只留於口頭。必須將 DoD 形成文字或清單，內建於使用者故事的生命週期中，用制度消滅「聽者誤解」與「說話者描述不準」的通病。
    
2. **交付時的「驚訝」是敏捷失敗的鐵證，真正的完工是理所當然**：高頻考點必須鎖定 **No Surprises** 原則。在評審會議（Review）上，客戶看到功能時的驚喜或困惑（「哇，我沒料到這個」），在敏捷眼裡都是未正確對齊 DoD 的管理警訊。完美建立 DoD 的專案，客戶看到成品時應覺得完全符合預期，不該有任何意外。
    
3. **DoD 必須橫跨使用者故事、發布到專案的三維架構**：完成的定義是一套立體的品質防禦體系。不能誤認為 DoD 只是工程師的代碼測試標準。在使用者故事（User Story）、版本發布（Release）與最終交付物（Final deliverables）上，都必須有對應的「Done」的定義，才能確保微觀的代碼品質與宏觀的商業戰略目標維持百分之百的一致性。
