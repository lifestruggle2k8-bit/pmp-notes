# 事前驗屍 (Pre-Mortems) 技術終極實戰指南

在敏捷風險管理與適應性規劃（Adaptive Planning）中，與其在專案死後才痛苦地開會撰寫驗屍報告，敏捷更推崇在專案開火前進行主動阻擊。**事前驗屍 (Pre-Mortems)** 是一種**在專案發生失敗之前，團隊共同坐下來預先尋找、挖掘並狙擊可能導致專案失敗因素的團隊協作活動**。

它強調「**Pre**」（事前）的預防性概念，與傳統專案在末尾才召開的事後檢討（Post-mortem）具有本質上的範式對比。

## 一、 Pre-Mortem 的三大標準執行流程

Pre-Mortem 是一場高規格的腦力激盪工作坊，由整個交付團隊共同參與，標準執行步驟如下：

```
 🎭 Pre-Mortem 標準化拆彈三大步驟：
 
  【1. 構思失敗情境】 ──► 在專案正式執行前，引導全員進入時空穿越想像：
                           「假設時間到了未來，而這個專案已經徹底失敗了，它會是什麼悲慘樣子？」
                                 │
                                 ▼
  【2. 彙整原因清單】 ──► 針對上述想像的慘劇，全員無情列出所有可能導致專案脫軌 (Derailed)
                           的潛在原因，並將這些原因進行跨維度的整合與彙整。
                                 │
                                 ▼
  【3. 審查計畫對策】 ──► 針對原因清單，重新回頭鐵腕檢視現有的專案計畫或 Product Backlog，
                           決定採取哪些具體行動來「減少」或「消除」這些失敗因子的發生機率。
```

### 🔍 步驟二的跨維度失敗原因探索（範例彙整）

團隊在此階段可以毫無禁忌地從多個臨床維度，列出讓專案崩盤的黑天鵝原因：

- 💻 **技術層面**：軟體開發嚴重延遲、核心系統崩潰、關鍵雲端服務全面停機。
    
- 👥 **人力層面**：核心專家可用性（Availability）出問題、甚至專案經理（PM）突然缺席或離職。
    
- 🤝 **外部層面**：利害關係人消極抵抗、或是客戶完全不願意參與（Active Participation）或配合。
    

## 二、 Pre-Mortem 的核心管理特性

這項技術看似天馬行空，實則具備極強的營運管理學防禦價值：

- **📊 與風險評估高度對齊**：它的底層邏輯與風險評估（Risk Assessment）非常相似，都是為了在危機發生前精確計算出風險的嚴重程度（Risk Severity）。
    
- **🛠️ Low-Tech, High-Touch 的典範**：它不需要配置任何高昂複雜的數位軟體，僅需白板與便利貼即可流暢運作。它高頻仰賴團隊成員之間深度、實質的面對面參與和情感交流（High-Touch），是催化隱性知識流動的絕佳工具。
    
- **🚀 最佳實施時機**：**強烈建議在專案正式啟動（Kick-off）或 Agile 專案正式執行之前進行！** 這種在期初將失敗因子顯性化的動作，能用最低的成本在關鍵時刻徹底挽救專案。
    

## 三、 🛠️ 跨章節深度聯想：如何將 Pre-Mortem 轉化為待辦清單的防禦燃料？

Pre-Mortem 絕對不是開完會、列完清單就結束的流於形式（Formality）。它產出的「具體減緩行動措施」，必須與我們先前學過的敏捷範疇工具進行剛性對齊。

當團隊在「步驟三」中審查計畫、制定出應對失敗原因的具體行動後（例如：為了防止雲端服務停機，決定準備一套替代函數或進行 PoC 技術驗證），PO 與 PM 必須將這些行動轉化為具體的 Spike 故事，直接插入 Product Backlog 中，轉化為 **風險調整後待辦清單 (Risk Adjusted Backlog)**！

$$\text{\textbf{Pre-Mortem 挖掘出技術風險} }\ \ \mathbf{\longrightarrow}\ \ \text{\textbf{制定出 Architectural / Risk-Based Spike} }\ \ \mathbf{\longrightarrow}\ \ \text{\textbf{插入 Backlog 頂端在下期迭代立刻擊穿}}$$

這套工法完美將「失敗的想像」降維轉化為「日常衝刺的剛性防護網」，徹底壓低了後期的**變更成本曲線**。

## 💡 三點需要牢記的部分

1. **Pre-Mortem 是在專案活著時進行「死後想像」，看到期初讓全員參與構思失敗情境立刻鎖定它**：這是極高頻的風險與團隊引導（Facilitation）情境考點。當題目提到「專案即將啟動，PM 想要帶領團隊全面識別隱性風險、打破成員不敢談論失敗的組織政治僵局」，**最佳的 Low-tech, high-touch 工具就是 Pre-Mortem**。它透過顛覆性的逆向心理學（假設專案已經死了），能強力突破人類的保守式失敗（Preferring to fail conservatively）盲區，引導出最真實的風險因子。
    
2. **挖掘出風險只是上半場，重新審查計畫並實施「變更」才是消滅脫軌（Derail）的硬道理**：必須精確鎖定 Pre-Mortem 執行流程的第三步核心。如果開完會只留下一張「經理可能離職、客戶不配合」的失敗原因清單，這屬於瀑布流的教條官僚浪費。敏捷要求必須**重新審查專案計畫（Review project plan）**，透過調整範疇、增加安全步驟或修改 Ground Rules 來減少或消除風險。任何沒有引導到「計畫修正與行動落地」的 Pre-Mortem 選項皆是虛假答題陷阱。
    
3. **用 Pre-Mortem 發揮僕人式領導的防護主權，把被動的「事後驗屍」前置升級為「主動獵殺」**：請將本篇概念與**持續改善（Kaizen）**、變更成本（Cost of Change）進行跨維度聯想。在 Production 階段或專案末尾才發現缺陷（Escaped Defects），其修復代價是滅頂之災。PM 作為僕人式領導者，在專案啟動前發起 Pre-Mortem，本質上是在縮短不確定性的回饋週期。用一場高互動的工作坊，把技術、人力與外部合規風險阻擊在第一線，這才是讓自組織團隊歷史速率（Velocity）永遠處於綠區安全範圍的最高防禦智慧。
