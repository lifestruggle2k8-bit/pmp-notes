# 參與利害關係人 (Engaging Stakeholders) 核心指南

專案本質上**是由人所做，也是為了人而做 (Projects are done by people, for people)**。因此，在敏捷管理中，如何透過制度與人際技巧維持利害關係人的高度參與，是決定專案成敗的關鍵。

## 一、 透過短迭代與定期發佈維持參與度

敏捷專案拒絕閉門造車，而是透過高頻率的節奏讓利害關係人保持視覺與體感上的連結：

- **時間盒限制 (Timeboxing)**：敏捷強調使用短迭代（Short iterations）與定期發佈（Releases），且**每個迭代週期嚴格規定不應超過四週**。
    
- **每月至少一次的進度感知**：在每個迭代結束時，利害關係人會被邀請親自參與**產品審查會議（Sprint/Iteration Review）**。這能確保他們**至少每月一次**，可以親眼看到、甚至親自操作目前的進度，真實感知產品的呈現方式與感受，確保團隊沒有偏離軌道。
    
- **軟體發佈的實體回饋**：每當有新版軟體交付與發佈時，利害關係人便能立即開始實際使用。讓使用者在真實營運環境中操作以**立即獲取回饋**，正是敏捷交付的核心目標所在。
    

## 二、 頻繁回饋的雙面效益：提早辨識風險 vs. 變更請求

讓利害關係人太早且太頻繁地測試產品，會帶來非常直觀的雙重效應：

### 1. 🌟 核心優點：提早識別風險與問題

- 透過讓利害關係人提早實際操作，能**在專案最終完成前，及早發現並修復各種潛在瑕疵**（例如：發現報告中缺少關鍵欄位、計算邏輯錯誤、或是遺漏了特定業務數據）。這能避免錯誤被埋藏到專案結束才爆發，大幅降低變更成本。
    

### 2. ⚡ 必然的副作用：導致更多的變更請求 (Change Requests)

- 實務上，利害關係人實際使用的頻率越高、玩得越深入，他們腦中衍生的新想法就越多，進而**提出更多的新需求與功能變更請求**。
    

### ☯️ 敏捷的正面轉念觀點

- 傳統專案常將變更請求視為控管不當的麻煩；但在敏捷原則中，**變更請求是利害關係人深度參與（Engagement）的健康表現！**
    
- 利害關係人還想要提變更，代表他們正積極投入並關注這個產品。透過這種動態調整，才能真正確保產品最終交付時，能百分之百符合他們的實際市場需求。
    

## 三、 處理解決利害關係人帶來的問題 (Issues)

只要與人合作，專案中就必然會預期遇到各種形形色色的衝突與問題（例如：利害關係人對專案經理的管理風格有意見、或是對特定功能的開發進度感到不滿意）。

```
 🛠️ 利害關係人問題 (Issues) 處理解法：
 
 ┌────────────────────────┐        ┌────────────────────────┐
 │   專案經理人際交往技巧   │ ───►   │   建立「問題呈報流程」   │
 │  (Interpersonal Skills) │        │  (Escalating Process)  │
 └────────────────────────┘        └────────────────────────┘
             │                                 │
             ▼                                 ▼
   第一線面對面溝通、同理         當問題超出 PM 權限或僵持時
   化解日常不滿與認知偏差         有清晰的合規路徑向上尋求裁決
```

- **軟實力的應用**：敏捷專案經理（PM/Scrum Master）需要高度運用**人際交往技巧（Interpersonal skills）**，透過同理心、積極傾聽與談判來化解日常的不滿。
    
- **建立「問題呈報/升級流程」(Escalating Process)**：人際技巧並非萬靈丹。專案必須預先建立一套清晰透明的问题呈報與升級機制。當利害關係人的衝突或問題超出專案經理的權限範圍、或雙方陷入無法化解的僵局時，能夠有法定合規的路徑將問題妥善升級向上呈報，確保問題得到及時且適當的裁決。
    

## 💡 三點需要牢記的部分

1. **四週是迭代的法律上限，Review 是每月一次的航道對齊**：在備考情境題時必須鎖定數字鐵律：**一個迭代絕對不能超過四週**。保持短迭代的目的，是為了強迫執行 Iteration Review，讓利害關係人維持「至少每月一次」的頻率親自審查產品。不讓利害關係人看產品的敏捷專案，都是偽敏捷。
    
2. **變更請求不是找麻煩，而是高參與度的「商業勛章」**：必須徹底翻轉傳統 PM 敵視變更請求的心態。敏捷認為 **Change requests are signs of engagement**（變更請求是參與度的表現）。利害關係人提出變更，說明他們正在積極思考如何讓產品更好。PM 應引導 PO 將這些變更納入 Backlog 重新排序，以此作為確保產品最終符合市場真實需求的養分。
    
3. **人際技巧是第一道防線，呈報升級流程是專案的法定安全網**：面對利害關係人對進度或 PM 本身的 Issues，敏捷經理首先必須展現人際交往技巧（Interpersonal skills）進行僕人式的溝通與協調；但同時，系統必須內建 **Escalating process（問題呈報/升級流程）**。當遇到權限外或不可調和的利害關係人衝突時，果斷且合規地啟動升級機制，才是對專案健康度最理性的保護。
