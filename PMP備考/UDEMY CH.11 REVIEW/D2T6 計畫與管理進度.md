# ECO 流程領域 Task 6：計畫與管理進度 (Plan and Manage Schedule) 終極實戰指南

根據專案的權威定義，專案本質上是一項「暫時性的努力 (Temporary endeavor)」。因為專案具備明確的終點，絕不能無限期地持續消耗資源，所以必須進行極其嚴格的時程管理。時程管理是專案管理最核心的基本功 (Project Management 101)，其重要性與管理成本 (Cost) 完全平起平坐。

進度管理的核心戰略目標，在於**精確識別活動、排序邏輯、預估時長、並依據所選的方法論（Agile 或 Predictive）建立時程基準，在執行期鐵腕追蹤並管理所有進度變異。**

## 一、 📊 預測型進度管理（Predictive）的六大標準生命週期

在傳統預測型專案管理中，從精細的 WBS 工作包到最終的時程表輸出，必須嚴格依循以下環環相扣的標準化推進流程：

```
 🗺️ 預測型進度管理六大核心工作流程：
 
  【1. 計畫進度管理】 ──► 頂層設計：決定如何進行排程、如何定義活動以及如何確保專案按計畫進行。
           │
           ▼
  【2. 定 義 活 動】 ──► 將工作包 (Work Packages) 進一步分解為可排程的【具體活動 (Activities / Tasks)】。
           │
           ▼
  【3. 排 序 活 動】 ──► 確定活動之間的先後邏輯關係與順序，透過排序進而找出【關鍵路徑 (Critical Path)】。
           │
           ▼
  【4. 估算活動時長】 ──► 為每項活動分配預計所需的時間（常用方法：由下而上估算、專家判斷）。
           │
           ▼
  【5. 開發進度表】   ──► 整合所有資訊，選擇適合排程方法論（如 CPM 關鍵路徑法法或敏捷排程），建立完整時程圖。
           │
           ▼
  【6. 監控與控制】   ──► 在專案執行期間動態追蹤實際進度，確保實際執行符合預期計畫，將偏差降到最低。
```

> 📐 **分解活動 (Define Activities) 臨床實務比喻**：
> 
> 工作包（粉刷房間） $\rightarrow$ 分解為可排程具體活動：$\text{【去商店】} \rightarrow \text{【買油漆】} \rightarrow \text{【回家】} \rightarrow \text{【粉刷牆壁】}$。在 PMP 考試語境中，**活動 (Activity) 與任務 (Task) 通常指涉相同概念，可完全互換使用**。
> 
> 估算專案任務時，必須剛性考量三大核心要素：**里程碑 (Milestones)、依賴關係 (Dependencies) 以及敏捷的故事點 (Story points)**。

## 二、 ⚖️ 方法論對齊：時程三角形的「翻轉邏輯」與變更治理

專案經理人必須根據專案所選定的方法論，採取黑白分明的時程準備（Prepare）、衡量（Measure）與修改（Modify）戰術。

### 1. 🧱 傳統預測型專案 (Traditional Project) ── 剛性時程基準

- **衡量與控制**：其時程表的結構一經核准即形成「時程基準（Schedule Baseline）」。
    
- **修正路徑（Change Request）**：在執行期一旦發現進度變異或面臨任何時程變更，**通常需要無條件透過正式的「變更請求 (Change Request)」程序提交 CCB 審查。** 變更時還必須高瞻遠矚地考量並協調與其他專案或營運單位之間的組織級制約。
    

### 2. 🟢 敏捷適應型專案 (Agile Project) ── 翻轉的三角形原則 (Project Triangle)

敏捷專案**並非擁有一個死板、固定的長週期時程表**，而是試圖在既定的固定迭代時間框架（Timebox）內完成盡可能多的工作。這屬於管理學上核心的**三角形原則翻轉**：

```
   🧱 傳統預測型三角 (Scope Fixed)                  🟢 敏捷適應型三角 (Time & Cost Fixed)
   
          【 範 疇 (固定) 】                             【 時間 】 ── 【 成本 】
             /          \                                   \          /
            /            \                                   \        /
     【時間】 ─── 【成本】 (變動)                           【 範 疇 (動態變動) 】
```

- **敏捷時程精髓**：因為**時間與成本相對固定**（例如一個 Sprint 就是固定兩週、固定成本），**範疇是動態變動的**。團隊依據歷史速率（Velocity）在固定時間盒內交付最高價值的產品待辦清單（Product Backlog）增量。
    

## 三、 🌐 組織級清障：跨專案資源協調 (Coordinate Resources)

在專案推進的過程中，會高頻涉及跨專案、跨組織階段的資源利用。**專案經理必須保持高度的「主動性 (Proactive)」，與其他專案經理或營運單位 (Operations) 進行高頻協調，以確保關鍵資源能準時到位**，杜絕資源爭用（Resource Contention）引發時程脫軌。

> 🚨 **跨項目資源爭用 (Resource Contention) 臨床情境診斷**：
> 
> 你的跨職能研發團隊即將完成代碼開發，下一步急需測試人員（QA）介入進行系統測試。然而，該測試人員此時可能正被指派支援平行宇宙中的另一個專案、或是隸屬於功能單位的營運資源庫（Operations）。

- **優秀 PM 的標準提前引導動作**：
    
    PM 絕不能等到代碼寫完、測試人員沒來才去開會抱怨（這是資訊滯後）。**必須提前 3 到 4 天與平行專案經理或功能經理進行剛性協調**：_「我的專案預計在 3 到 4 天後釋出增量，我需要在既定時間內使用測試人員，請確認其可用性（Availability）。」_ 確保組織資源能無縫銜接。
    

## 💡 三點需要牢記的部分

1. **傳統專案變更時程必須走「Change Request（變更請求）」，看到 PM 發現進度落後就私自改動時程表的選項請鐵腕秒殺**：這是 Process 領域進度控制中極高頻的情境題題眼！只要題目情境明確界定在預測型專案（Traditional），當執行期發生任何時程偏離、或客戶提出時程調整時，**請堅決無視任何『由 PM 評估後直接重新排程（Reschedule）』的獨裁選項**。只要動到進度基準，唯一的合規綠區就是提交正式的 Change Request 走變更控制流程。
    
2. **敏捷專案的「時間與成本固定、範疇變動」，看到敏捷團隊為了塞入新需求而延長 Sprint 週期或大幅加班的選項請判定為嚴重違反敏捷宣言**：必須死死刻死敏捷排程的翻轉三角形原則。在備考情境題時，如果題目提到「在敏捷專案中，PO 突然在衝刺中期塞入高優先級功能，團隊評估做不完，問 PM 如何處理」，**正確答案絕對不是延長兩週的固定 Timebox，更不是砍掉測試時間**。敏捷的時間是剛性卡死的，做不完的範疇故事應在 Sprint Review 後重新退回 Product Backlog 重新排序。分清敏捷三角形的邊界，才能精準擊穿考場陷阱。
    
3. **跨專案資源爭用（Resource Contention）必須「提前 Proactive 協調」，嚴禁在關鍵路徑卡死、人員沒到場時才向高層打小報告**：請將本篇的資源協調，與我們先前學過的**處理妨礙與障礙（Address Impediments）角色參與度分析**、以及**談判專案協議（Negotiate Project Agreements）功能經理爭取人力**進行深度跨章節橫向聯想。當測試人員或網路工程師在營運單位（Operations）與多個項目間流轉時，PM 作為**僕人式領導者（Servant Leader）**，必須用最高規格的透明度（Information Radiator）提前預判依賴關係（Dependencies）。提早與平行 PM 達成資源調配協議，專案才能在健康、安全的軌道上順利衝線。
