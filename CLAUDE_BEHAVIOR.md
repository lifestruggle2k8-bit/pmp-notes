# CLAUDE.md — 專案管理輕鬆學 Wiki 行為準則

基於 Andrej Karpathy 的 LLM 編碼最佳實踐改編

---

## 四大原則

### 1️⃣ Think Before Coding — 思考優先

**規則：提出問題，不要猜測**

#### 術語頁面新增時
- [ ] 先檢查是否已有同義詞（如：PMP vs 國際專案管理師）
- [ ] 確認該術語在 all_pages.txt 中的上下文
- [ ] 如果定義有歧義，先問而非猜測

#### 章節摘要修改時
- [ ] 列出將影響的相關章節（3個以上才修改）
- [ ] 檢查雙向連結是否需要更新
- [ ] 如果涉及跨章節概念，先徵詢用戶意見

#### 不確定時停止
```
❌ 錯：直接生成術語定義，猜測 Gemini API 回應
✅ 對：先列出3個可能的理解，問用戶是哪個
```

---

### 2️⃣ Simplicity First — 簡約第一

**規則：最小化即可解決問題，不要過度設計**

#### 術語頁面格式（精簡版）
```markdown
---
title: "{術語}"
type: term
category: "{類別}"
tags: [專案管理術語, {類別}]
created: 2026-04-17
---

# {術語}

## 摘要
{30-50字定義}

## 考試重點
- {重點1}
- {重點2}

## 相關術語
[[連結1]] [[連結2]]
```

#### 新增功能前自問
- 是否真的需要？
- 用戶有要求嗎？
- 能不能用現有結構滿足？

#### 清理規則
- 刪除未被引用的術語頁面（>1個月未更新）
- 移除 index.md 中的重複條目
- 合併同義詞（保留一個，其他設重定向）

---

### 3️⃣ Surgical Changes — 精確編輯

**規則：只改變必要的部分，其他不動**

#### 新增術語時
```
✅ 執行：
  1. 建立 .md 檔案到 wiki/concepts/terms/
  2. 更新 index.md 該術語對應的類別行
  3. 在 log.md 記錄新增

❌ 不執行：
  - 重新格式化其他術語頁面
  - 修改其他頁面的連結格式
  - 「改進」相關的舊內容
```

#### 更新章節摘要時
```
✅ 允許：
  - 新增引用
  - 修正錯誤事實
  - 更新相關連結

❌ 禁止：
  - 改變頁面結構
  - 重寫現有段落（除非確實有誤）
  - 移除舊連結（除非連結對象被刪除）
```

#### diff 檢查原則
每次修改前問：
- 這個改動直接回應用戶需求嗎？
- 有無關的改動被包含？
- 會影響其他頁面嗎？

---

### 4️⃣ Goal-Driven Execution — 目標驅動

**規則：轉換模糊需求為可驗證的成功標準**

#### 術語補齊任務
```
❌ 模糊：「補齊所有術語」

✅ 清晰：
  目標：為 all_pages.txt 中出現 3+ 次的術語建立頁面
  驗證標準：
    - 共 X 個術語待補
    - 每個術語頁面格式檢查：
      ☐ title 正確
      ☐ category 分類正確
      ☐ 摘要 30-50 字
      ☐ 相關術語連結存在
    - index.md 自動檢查：無孤立頁面
  完成標誌：執行檢查腳本無錯誤
```

#### 品質提升任務
```
❌ 模糊：「改進頁面品質」

✅ 清晰：
  目標：為所有術語頁面新增「考試重點」section
  驗證標準：
    - 掃描 wiki/concepts/terms/ 下所有 .md 檔
    - 檢查每個檔是否含有 ## 考試重點
    - 每個重點 1-2 句，至少 3 個
  完成標誌：掃描返回 0 個缺陷
```

#### 每次執行寫驗證清單
```
任務：新增 50 個敏捷相關術語
計畫：
  Step 1: 生成術語定義 → 驗證：Gemini API 成功率 >95%
  Step 2: 建立 .md 檔案 → 驗證：50 個檔案已建立
  Step 3: 更新 index.md → 驗證：index.md 可解析，無語法錯誤
  Step 4: 掃描孤立連結 → 驗證：無未定義連結

成功的標誌：4 個驗證都通過
```

## 實施指南

### 術語頁面標準檢查清單
```markdown
- [ ] title 與檔名一致
- [ ] category 從預設列表選（不創建新類別）
- [ ] 摘要 30-50 字（中英文合計）
- [ ] 相關術語連結數 ≥2
- [ ] 無拼寫錯誤
- [ ] 無孤立連結（[[X]] 但沒有 X.md）
```

### 修改前停頓清單
```
執行修改前，問自己：
- [ ] 用戶明確要求了嗎？
- [ ] 我確定這是正確理解嗎？
- [ ] 會不會同時改動無關的東西？
- [ ] 改完後怎麼驗證成功？
```

## 常見陷阱

| 陷阱 | 症狀 | 修正 |
|------|------|------|
| **過度設計** | 每個術語頁面都有10個section | 只保留：摘要 + 考試重點 + 相關術語 |
| **隱藏假設** | 建立術語後才發現定義有歧義 | 先驗證定義，再建頁面 |
| **副作用修改** | 修改一個術語導致其他連結破裂 | 修改前先查詢 grep 的引用數 |
| **無驗證執行** | 新增100個術語，不知道成功率 | 寫驗證標準在計畫中 |

---

## 記住

> **LLM 在有清晰目標和驗證標準時表現最好**

不是：「幫我加術語」
而是：「建立出現3+次的術語，每頁有摘要+重點+連結，完成後執行 check_orphans.py」

差別在於後者可以自動驗證、可以中斷重試、可以衡量進度。
