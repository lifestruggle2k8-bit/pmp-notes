# 階段與可交付成果 (Phases and Deliverables)

專案是由多個階段（phases）組成，每個階段產出特定的可交付成果（deliverables）。理解階段和可交付成果的概念是進行專案規劃和監控的基礎。

## 📖 原文內容

### 階段的定義

**階段**是專案中的一個邏輯序列，通常在一個主要的里程碑時結束。每個階段都有特定的目標和可交付成果。

### 可交付成果的定義

**可交付成果**是過程或階段的有形或無形結果，通常必須經過批准才能被視為完整。

### 階段的特點

| 特點 | 說明 |
|------|------|
| **順序性** | 階段按邏輯順序進行，通常是順序的 |
| **有明確邊界** | 每個階段有開始和結束 |
| **獨立審批** | 每個階段完成時進行階段審批 |
| **可重疊** | 某些方法中階段可能重疊（如敏捷） |
| **里程碑標記** | 階段結束時達到里程碑 |

### 可交付成果的類型

| 類型 | 說明 | 例子 |
|------|------|------|
| **過程性交付成果** | 支持專案管理的文件和計畫 | 專案計畫書、需求文件 |
| **產品性交付成果** | 專案的最終產品或服務 | 軟體、報告、設施 |
| **組合性交付成果** | 多個專案或階段的組合結果 | 系統集成、總體方案 |

### 階段vs完整專案

```
傳統瀑布式：
階段1 → 階段2 → 階段3 → 階段4 → 完成

敏捷迭代：
迭代1 → 迭代2 → 迭代3 → 迭代N → 完成
(每個迭代產出遞增的可交付成果)
```

---

## 🎯 重點摘要

- **階段是結構化的**：將專案分解為可管理的序列。

- **每個階段有可交付成果**：階段的完成以可交付成果的交付為標誌。

- **階段審批很重要**：在進入下一階段前，應對可交付成果進行審批和確認。

- **可交付成果必須完整**：必須達到質量標準和驗收準則。

- **靈活方法可能改變階段結構**：敏捷方法中迭代替代傳統階段。

## 📋 考試要點

**常見題型：**
- ✅ 定義題：「什麼是可交付成果？」
- ✅ 區別題：「過程性交付成果與產品性交付成果的區別」
- ✅ 應用題：「識別某階段的可交付成果」

**記憶技巧：**
- 🧠 **可交付成果=具體產出物**：必須有形或無形但可驗證
- 🧠 **階段=邏輯序列**：每個階段應有明確的目標和邊界
